# Project Guardrails
> **Agent Rule**: Read this file at PHASE 0 and re-check at PHASE 5 (Guardrail Check) before producing any output.

---

## 1. MUST-DO Rules (Non-Negotiable)

| # | Rule | Enforcement |
|---|---|---|
| G1 | Every `it()` callback MUST be `async function ()` — never an arrow function `() =>` | Hard block |
| G2 | Every LeanFT SDK call MUST be `await`-ed — no fire-and-forget | Hard block |
| G3 | Always call `runner.setup()` in `beforeAll`, `runner.teardown()` in `afterAll` | Hard block |
| G4 | Always call `runner.beforeEachTest()` in `beforeEach`, `runner.afterEachTest()` in `afterEach` | Hard block |
| G5 | Never hardcode credentials, account numbers, or PII in spec files | Hard block — use `td.get()` |
| G6 | Never use `sleep()` or `setTimeout()` for waiting — use `waitForScreen()` only | Hard block |
| G7 | Never import from a higher layer than your own layer — specs only import from libraries/driver/config/testdata/objectrepository | Hard block |
| G8 | `reporter.startTestCase()` must be the first call inside every `it()` body | Hard block |
| G9 | Never log, screenshot, or print the PASSWORD field or its value | Hard block — security |
| G10 | All files must start with `"use strict";` | Hard block |

---

## 2. SHOULD-DO Rules (Strong Preference)

| # | Rule |
|---|---|
| S1 | Use composite `terminal.login()`, `terminal.signOff()` etc. when available — don't re-implement step-by-step in the spec |
| S2 | Use `reporter.logStep()` before each logical block of steps for traceability |
| S3 | Use `reporter.logSnapshot()` after navigating to a new screen (except LOGON_SCREEN) |
| S4 | Reference test data via `td.get("DataKey")` — never inline the values |
| S5 | Screen IDs used in `waitForScreen()` must exactly match keys defined in `objectrepository/screens.js` |
| S6 | Assertions MUST use `expect(actual).toBe(expected)` from `leanft/expect` — not raw `if` statements |
| S7 | Each `it()` block should test ONE logical scenario — do not combine unrelated assertions |
| S8 | `jasmine.DEFAULT_TIMEOUT_INTERVAL` must be set at the top of each `describe` block using `settings.execution.defaultTimeout` |

---

## 3. MUST-NOT-DO Rules (Anti-Patterns)

| # | Anti-Pattern | Reason |
|---|---|---|
| A1 | Do NOT use `done()` callback style — always use `async/await` | Deprecated in Jasmine 4+; LeanFT SDK is Promise-based |
| A2 | Do NOT use `browser` or `page` objects | This is a 3270 TE framework, not a web framework |
| A3 | Do NOT import `leanft` directly in spec files for SDK init — use `driver/runner.js` | Single responsibility |
| A4 | Do NOT use `.then().catch()` chains — always `async/await` with try/catch if needed | Readability |
| A5 | Do NOT use `require()` paths that skip a layer (e.g. spec directly importing `leanft.sdk.te`) | Layer violation |
| A6 | Do NOT put field `attachedText` values inline in spec files — always via `objectrepository/screens.js` | Maintainability |
| A7 | Do NOT omit the `"use strict"` directive | Enforcement of strict mode |
| A8 | Do NOT use `var` — use `const` or `let` | Modern JS standard |
| A9 | Do NOT silently swallow errors with empty `catch {}` blocks | Causes false positives in test results |
| A10 | Do NOT use `console.log` for reporting — use `reporter.logStep()` | LeanFT report integration |

## 4. Migration-Specific Guardrails

| # | Rule |
|---|---|
| M1 | If a VBScript keyword has no direct equivalent in `terminalHelper.js`, emit a `// TODO: MANUAL_REQUIRED – <reason>` comment and do NOT invent code |
| M2 | If a screen or field is referenced in VBScript but NOT defined in `objectrepository/screens.js`, emit `// LOCATOR_MANUAL_REQUIRED – add <SCREEN_ID>.<fieldName> to screens.js after OIC capture` |
| M3 | If test data referenced in VBScript is NOT in `testdata/ftd_testdata.json`, emit `// TESTDATA_MANUAL_REQUIRED – add key "<key>" to ftd_testdata.json` |
| M4 | Never guess `attachedText` values — they must come from the Object Repository or be flagged as `LOCATOR_MANUAL_REQUIRED` |
| M5 | Preserve the original VBScript comment / test case name as a `// [Migrated from VBScript] <original name>` comment above the `it()` block |
| M6 | Each migrated `it()` block must have a corresponding entry in the Migration Delta Report |

## 5. File & Folder Conventions

```
spec/                         ← All spec files go here
  <feature>_jasmine_spec.js
libraries/                    ← Shared keyword libraries only (no test logic)
objectrepository/screens.js   ← Single file — extend with new screens, do not create new files
testdata/ftd_testdata.json    ← Single file — add new keys, do not create new JSON files
config/settings.js            ← Single file — no duplication
results/                      ← Auto-generated by LeanFT — never commit content
```

---

## 6. CI/CD Guardrails

- `Jenkinsfile` is the only pipeline definition — do not add `.github/workflows` YAML pipeline triggers to this repo
- Environment is controlled by `TEST_ENV` env var (`SIT` or `UAT`) — never hardcode environment in spec code
- `TE_SHORT_NAME` is injected by Jenkins — spec files must read it via `settings.terminal.shortName`
