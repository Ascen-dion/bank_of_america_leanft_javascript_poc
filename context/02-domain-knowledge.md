## 1. Business Domain – Fixed Term Deposits (FTD)

The GBS mainframe application manages core banking operations. The POC focuses on the **FTD (Fixed Term Deposit)** domain.

| Term | Definition |
|---|---|
| **FTD** | Fixed Term Deposit — a savings product where a customer deposits a fixed amount for a defined term at a fixed interest rate |
| **GBS** | Global Banking System — the IBM mainframe application hosting core banking |
| **SIT** | System Integration Testing environment |
| **UAT** | User Acceptance Testing environment |
| **FTD_Login** | The test dataset key for logon credentials (userId, password) |
| **AccountInquiry** | The test dataset key for account inquiry test data |
| **FundsTransfer** | The test dataset key for funds transfer test data |

## 2. Core Business Workflows (Test Scenarios)

### 2.1 FTD Login
**Purpose**: Authenticate a user against the GBS mainframe.

**Steps**:
1. Navigate to `LOGON_SCREEN`
2. Enter `userId` in the USER ID field
3. Enter `password` in the PASSWORD field (masked — do not log value)
4. Press Enter
5. Verify `MAIN_MENU` appears

**Business rules**:
- Passwords are case-sensitive on GBS
- Three failed login attempts lock the account
- Never log or screenshot the PASSWORD field value

### 2.2 Account Inquiry
**Purpose**: Look up a customer account and verify balance and status.

**Steps**:
1. From `MAIN_MENU`, enter option `01` and press Enter
2. On `ACCOUNT_INQUIRY`, enter account number
3. Press Enter to retrieve account data
4. Assert: account holder name, balance, and status match expected test data

**Business rules**:
- Account numbers are 10-digit numeric strings (zero-padded)
- Status values: `ACTIVE`, `DORMANT`, `CLOSED`
- Balance is displayed as a formatted number with 2 decimal places (e.g. `1,234.56`)

### 2.3 Funds Transfer
**Purpose**: Transfer funds between two accounts.

**Steps**:
1. From `MAIN_MENU`, enter option `02` and press Enter
2. On `FUNDS_TRANSFER`, enter: from account, to account, amount, currency, reference
3. Press Enter to submit
4. Assert: confirmation screen appears with a transaction reference number
5. Press PF3 to return to `MAIN_MENU`

**Business rules**:
- Amount must be a positive value with up to 2 decimal places
- `FROM ACCOUNT` and `TO ACCOUNT` must differ
- Currency is a 3-letter ISO code (e.g. `USD`, `GBP`)
- Transaction reference is generated by GBS — capture it for reporting

### 2.4 Sign Off
**Purpose**: Safely terminate the GBS session.

**Steps**:
1. From `MAIN_MENU`, enter option `99` and press Enter
2. Verify `LOGON_SCREEN` reappears (session clean)

## 3. Test Data Structure

Test data is stored in `testdata/ftd_testdata.json` and loaded via `testdata/testDataManager.js`.

```json
{
  "FTD_Login": {
    "userId": "...",
    "password": "..."
  },
  "AccountInquiry": {
    "accountNumber": "...",
    "expectedName": "...",
    "expectedBalance": "...",
    "expectedStatus": "ACTIVE"
  },
  "FundsTransfer": {
    "fromAccount": "...",
    "toAccount": "...",
    "amount": "...",
    "currency": "USD",
    "reference": "..."
  }
}
```

**Agent rule**: Test data keys (`FTD_Login`, `AccountInquiry`, `FundsTransfer`) are the canonical names. Use them exactly as-is when referencing test data in generated code. Use `td.get("FTD_Login")` syntax.

## 4. Assertion Patterns

| What to assert | How |
|---|---|
| Screen reached | `await terminal.waitForScreen(win, "SCREEN_NAME")` — throws on timeout |
| Field display value | `const val = await terminal.getFieldValue(win, "SCREEN_NAME", "fieldName"); expect(val).toBe(expected)` |
| Transaction reference captured | Store in local variable; include in `reporter.logStep` |
| No error on screen | Implied by successful `waitForScreen` (navigation succeeded) |

---

## 5. Sensitive Data Rules

- **Never** include real credentials in test data files committed to Git
- **Never** log, screenshot, or print the `PASSWORD` field
- **Always** read credentials from test data file (`ftd_testdata.json`) — never hardcode in spec files
- The `reportHelper.logSnapshot()` call must be suppressed or blurred when the current screen is `LOGON_SCREEN`
